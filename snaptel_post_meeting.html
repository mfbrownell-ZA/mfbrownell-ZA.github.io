Risk Flags — New or Elevated Since the Call

Single developer dependency (Tiaan)

The entire technical build sits with Tiaan. If something goes wrong post-launch and Tiaan is unavailable, TickeyBox has no ability to intervene independently. Hannes mentioned ~6 developers for support, but Tiaan is the core engine person. This must be explicitly addressed in the SLA — defined escalation paths, named backups, and response time commitments that don't rely on a single individual.

IP ownership ambiguity creates provider lock-in

If the app and backend are built entirely by SnapTel and IP ownership is not formally agreed, TickeyBox has no leverage in a commercial dispute. Switching providers becomes extremely expensive — not just migration cost, but rebuilding the entire consumer-facing layer. This needs to be resolved in writing before going live, not after.

Equity conversation without a framework

Hannes raised equity as a concept without defining it. If this drifts — through informal conversations and deepening technical dependency — TickeyBox could find itself in a de facto partnership with undefined terms. The right move is to have a deliberate commercial conversation that either closes the equity discussion (fee model only) or structures it properly (equity stake, vesting, IP terms, exit).

Competitive threat — someone else is circling

Grandstream has flagged an unidentified party exploring the same product space. SnapTel can deploy infrastructure quickly, but the front-end experience (app, provisioning UX) is the differentiator and the thing that takes time. Closing the dogfooding phase and getting to a live pilot as fast as possible is now more urgent than previously framed.

What Was Confirmed — Clean Closes

Architecture is solid and future-proof

The per-household PBX model is not a technical dead-end. It's virtual segmentation on shared infrastructure, fully scalable, and allows failover between physical hardware. No redesign required at any forecast volume. This removes one of the biggest technical unknowns from the risk register.

Full API access confirmed — no lock-in at the integration layer

Every function is API-driven. Webhooks on every event. TickeyBox can build its own logic, app, and integrations against the SnapTel API without being forced through SnapTel's proprietary UI. This preserves long-term optionality even if the equity conversation doesn't go anywhere.

Free internal calling is straightforward to implement

The network effect mechanism (TickeyBox-to-TickeyBox free) requires no complex infrastructure work. SnapTel knows which numbers are on-platform and can rate them at zero. This is a strategic product feature that costs nothing to implement.

Prepaid billing alignment — no model changes required

SnapTel's prepaid credit model for resellers maps directly onto TickeyBox's end-user prepaid minute model. Working capital requirements are unchanged. International calling excluded by default — exactly what TickeyBox wants.

Build is closer to completion than expected

Development is approximately one week from completion. Dogfooding with 9 households can begin as soon as the build is approved. This accelerates the path to pilot significantly and means the April pilot timeline is more achievable than it appeared before the call.

Waitlist above 300 — strong context setter

Opening with 300+ waitlist entries (all organic) set the tone. Hannes heard that there is real demand and a partner worth investing in. This likely contributed to the openness around equity and the willingness to have the dev build already underway.
