Traction
Where TickeyBox Stands Right Now
All growth to date has been organic — no paid acquisition, no influencer activity. The 325-household waitlist is the product of a monthly community newsletter and word-of-mouth sharing.
📋 The waitlist is growing in pods, not one-by-ones

People are signing up in friendship groups of 5–10 households. A single WhatsApp share on 23 March added 20 households in a day. The driver is almost always the same: someone discovers TickeyBox-to-TickeyBox calls are free, then pulls their friend group in so they can call each other. The network effect is already happening before the product exists.

🌍 International demand is appearing

More than five separate inbound enquiries have come from parents asking whether grandparents in the UK or Australia can get a TickeyBox — primarily divorced families wanting the remote parent to have a simple, safe way to call. As confirmed in the workshop, TickeyBox-to-TickeyBox calls work across any endpoint globally. International device sales are possible; the subscription pricing would reflect the local market. This is not a short-term priority, but it's worth noting the demand is organic and unsolicited.

📦 Hardware and supply chain is in good shape

TickeyBox is now an official reseller through First Distribution. 8 Grandstream GHP610WE units are in hand for the dogfood group. Grandstream are supporting the pilot hardware order (~50 units), with a backup supply route available. The 1,000-unit branded order — with custom TickeyBox colours, logo, and packaging — is in early discussions. Grandstream have confirmed they'll produce a custom run at 1,000 units. That order requires an 8–10 week lead time from placement, which sets the go/no-go timing for full launch.

💡 Why the Tin Can comparison matters

Tin Can (Seattle) is the proof of concept for this model globally. They're on their 5th batch drop — 10,000 units selling out weekly, now at 50,000+ across their network. Their model is simple: batch orders, 10-week delivery windows, and free device-to-device calls as the viral mechanic. TickeyBox is building the South African version of this, with the advantage of a live VoIP infrastructure partnership and a distribution setup that's already moving.

To Test
Device Provisioning — The Agreed Approach
This is the most technically uncertain part of the setup. The goal is a process that a parent with no technical background can complete without needing to call anyone. We're not there yet — but we have a clear path to test.
1
Warehouse pre-configuration

When an order comes in, there's a 3–4 day window before the phone needs to ship. In that window, TickeyBox creates the household record in the system, assigns a number (from DataFon, 061 prefix based on the registered address), and generates the FQDN for the PBX segment. The phone's MAC address is registered against this household. SIP credentials are ready before the box leaves the warehouse.

Pre-ship, no customer involvement
2
Parent unboxes and plugs in via LAN

Parent opens the box, plugs the phone into power and into the router via a LAN cable (a yellow cable is included in the box — make it obvious). The phone boots, contacts the provisioning server, and pulls its config. At this point the phone is live on the network. SIP registration happens automatically. The parent doesn't touch any settings.

LAN cable included in box. One plug, one cable.
3
WiFi setup via Grandstream Cloud API

The open question. The plan is to use the Grandstream Cloud portal API to push WiFi credentials to the phone remotely — the parent enters their home network name and password in the TickeyBox app, and it gets injected into the phone config without them needing to access any admin interface. This needs to be tested. Tiaan is taking ownership of this test.

Needs testing — Grandstream Cloud API Critical path for consumer-ready setup
4
Parent unplugs LAN cable — phone works on WiFi

Once WiFi credentials are pushed and verified, the parent unplugs the LAN cable. The phone reconnects via WiFi, re-registers with the PBX, and is fully operational. From the parent's perspective: plug in, wait, follow one screen in the app, unplug the cable. That's it.

Target end state — subject to WiFi API test result
5
Fallback: manual WiFi setup during dogfood

If the Grandstream Cloud API approach doesn't work cleanly, the fallback for the dogfood phase is manual setup — TickeyBox walks each household through the WiFi configuration step. This is fine for 8–10 households. The learning from those sessions will inform whether we need an alternative technical approach before the 40-household pilot.

Dogfood fallback only — not acceptable for pilot scale

1
Customer subscribes via the TickeyBox web app

Account creation → address entry (for number allocation and emergency routing) → hardware bundle selection → PayFast recurring subscription set up. R1 charge at signup establishes the billing relationship.

2
PayFast debits monthly on the subscription renewal date

PayFast handles the recurring charge and all retry logic. Standard payment failure rate in SA is 7–8% — PayFast attempts 3 times before flagging as failed. TickeyBox receives a webhook notification on each payment event.

3
Webhook fires → Zoho Books generates invoice

On successful payment, a webhook from PayFast triggers Zoho Books to create and send the customer invoice. Invoices are stored in Zoho. Upgrades and downgrades to minute packages also trigger the same flow.

4
Failed payment: manual suspension

If a payment fails after retries, TickeyBox is notified. Tiaan/Hannes manually suspends the PBX from the SnapTel reseller portal. TickeyBox handles the customer communication. Reactivation happens once the payment clears. This is manual for now — automation comes later as scale demands it.

Manual at pilot scale — acceptable

The Priority Between Now and April 7

Two things unlock everything else
Get the web interface to a state where a parent can navigate it. It doesn't need to be beautiful. It needs to work without hand-holding. Whitelist, quiet hours, contacts, call log. That's the core. If a parent can do those four things unassisted, dogfooding can start.
Get Matt into the test environment. The best feedback on what's missing will come from Matt walking through the app with fresh eyes. The sooner the test login is shared, the sooner useful iteration can happen — even ahead of the April 7 session.
Everything else — billing integration, auto-provisioning, minute package management, SLA structure — comes after dogfooding starts. The dogfood phase is where we learn what actually matters to a real parent. Don't over-build before that.

Parking Lot — Post-Pilot Conversations

🅿 Items to revisit once the pilot is running

The following came up during the workshop but were deliberately set aside. They're not urgent now — and trying to resolve them before the pilot would slow things down. We'll pick these up once there's live traction to anchor the conversation.
Long-term commercial structure between TickeyBox and SnapTel. Hannes mentioned the possibility of a deeper commercial arrangement as scale grows. This is a conversation worth having — but at the right time, with real numbers on the table. Post-pilot is the right moment.
SLA for ongoing support and uptime. Hannes confirmed a formal SLA structure is possible and makes sense as the customer base grows. The shape of it — response times, escalation paths, pricing — is better designed once we understand what the real support load looks like. Pilot data will tell us that.
Native mobile app. Web-first for now. Once the web experience is proven and stable, the native app is the natural next step. The Snapdial dialer is available as a soft phone option in the interim for any parent who specifically wants the app on their phone.
International market expansion. Device-to-device calls work globally already. A proper international offering — particularly for the UK and Australian expat market — would require adapted subscription pricing and potentially a separate entity structure. Worth exploring, not worth prioritising now.
CRM and customer management tooling. Zoho Books handles invoicing. PayFast handles billing. A proper CRM for managing households at scale will be needed — but not at 40 households. This can be a lightweight build later and doesn't need to be a priority now.
