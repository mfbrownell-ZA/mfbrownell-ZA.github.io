UI, Copy & UX Feedback

Combined review of the demo app ahead of the 50-household pilot. Includes ideal state screen mockups as visual reference alongside each section. Focus: language, information architecture, UX flow, safety defaults. Brand colour pass is separate.

Date27 March 2026
Apptickeyboxdemo.snaptel.io
Issues37 identified
AudienceParents (non-technical)

Scope

  • In scope: Copy, labels, IA, UX flow, safety defaults, parent-facing language
  • Out of scope: Brand colours, logo, typography
  • Not flagged: Emergency contacts UI, transcriptions — both stubbed. Emergency contacts spec is defined below (#12) for when they are built.
  • Priority logic: P1 = fix before pilot. P2 = fix before broad rollout. P3 = polish.

What's Working

  • End-to-end flow functional (create → pay → provision → manage)
  • Contacts list with inbound/outbound control
  • Quiet Hours and Weekly Schedule (inc. per-day time pickers)
  • Minutes purchase via PayFast (recurring + one-time)
  • Multi-account switching
  • Call log and call details modal

Naming Convention — Agreed Standard

Use "TickeyBox" as the noun for the physical device. Use plain language for all actions and states. User-inputted names (e.g. "Matt's Phone") are shown as-is — do not flag these as dev issues. Test: would a non-technical parent understand it in under 2 seconds?

#11Contact form — unnecessary fields, split name, and a critical permissions default bug
P1SafetyIAUX
Screen: Add New Contact (confirmed from screenshot)
Confirmed fields on current form: First Name * | Last Name | Email | Phone * | ▶ Address Information (Optional, collapsible) | Notes | Call Permissions: ☐ Allow Inbound Calls / ☐ Allow Outbound Calls

Three problems:
1. Unnecessary fields: Last Name, Email, Address Information, and Notes have no function on a child's phone contact. They add friction and look like a corporate CRM.
2. Split name: "First Name" + "Last Name" is wrong for this product. A parent adds "Gogo", not "Gogo" + surname. One "Name" field only.
3. Critical — Call Permissions default to OFF (unchecked checkboxes): The Call Permissions section sits at the bottom of the form, below a large Notes textarea. Any parent who fills in the name and number and saves without scrolling to the bottom will create a contact that can neither call in nor call out — silently broken. This is a functional bug disguised as a UX issue.
Strip the form to four elements only:
Current (confirmed)
First Name * Last Name Email Phone * ▶ Address Information (Optional) Notes [large textarea] Call Permissions ☐ Allow Inbound Calls ☐ Allow Outbound Calls [checkboxes — unchecked by default]
Recommended
Name * [placeholder: e.g. Gogo] Phone number * [placeholder: e.g. 072 555 0101] [hint below: SA numbers only, 10 digits, starting with 0] Can call this phone [toggle: ON] Matt can call them [toggle: ON] [Save contact]
Permissions change: Replace checkboxes with toggles. Default both to ON. A parent explicitly turning one off is a deliberate choice. A parent not finding the checkboxes is a silent failure that breaks the product's core function.

Number validation on save: Reject any number that does not match a 10-digit SA format starting with 0 (mobile: 06x/07x/08x, landline: 01x/02x/03x/04x/05x). Reject +27 prefix and any international format. Inline error: "Please enter a South African number starting with 0, e.g. 072 555 0101."

Contact limit: Cap at 20 contacts per phone. When limit is reached, disable "+ Add a contact" and show: "You've reached the 20-contact limit. Remove a contact to add a new one."
#12Emergency Contacts — spec for when feature is built
P1NewIASafety
Screen: Emergency Contacts (currently "coming soon")
Emergency Contacts are currently stubbed ("coming soon"). The feature needs a clear spec before it is built so the implementation matches the intended product behaviour. Currently emergency contacts and the main contact list are not differentiated in the UI.
Product spec for Emergency Contacts:

— Maximum 3 emergency contacts per phone.
— Emergency contacts are a separate list from the main contact list (max 20).
— Emergency contacts override quiet hours and weekly schedule — their calls always ring through regardless of time restrictions.
— Emergency contacts use the same SA number format rules as regular contacts (10 digits, starting with 0, no +27, no international).
— The "Emergency calls always get through" toggle in Quiet Hours (see #17) is the runtime control for this. It should default ON.

UI spec: Emergency Contacts should appear as a separate tile/section in the phone management screen, clearly distinct from Contacts. Recommended copy:
Current
Emergency contacts feature coming soon!
When built
Emergency Contacts Up to 3 people who can always reach this phone — even during quiet hours. [max 3, same number format rules]
